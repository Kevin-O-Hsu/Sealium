## 优化后的在线激活验证模块方案

### 1. 激活码生成（服务端）
- 激活码由服务端随机生成，采用高强度随机数（如128位），编码为字符串。
- 数据库记录：激活码、绑定机器码、激活时间、授权截止时间、授权功能位、激活状态（未激活/已激活）。

### 2. 密钥准备
- 服务端生成RSA密钥对（公钥`PubS`，私钥`PriS`）。
- 客户端（在代码虚拟化模块中）内置服务端公钥`PubS`，并为自身生成RSA密钥对（公钥`PubC`，私钥`PriC`），`PubC`随请求发送。

### 3. 激活流程（HTTPS传输）

#### 3.1 客户端构造请求
- 获取机器码`MachineCode`（硬件信息哈希）。
- 生成随机数`Nonce_C`（16字节）和当前时间戳`Timestamp`（Unix时间戳，秒级）。
- 构造明文请求体：
  ```json
  {
    "activation_code": "用户输入的激活码",
    "machine_code": "MachineCode",
    "timestamp": Timestamp,
    "nonce": Nonce_C,
    "client_pubkey": "PubC"
  }
  ```
- 使用服务端公钥`PubS`对上述明文进行RSA-OAEP加密，得到密文`EncReq`。
- 通过HTTPS将`EncReq`发送至服务端激活接口。

#### 3.2 服务端验证
- 使用私钥`PriS`解密`EncReq`，得到请求明文。
- **时间戳校验**：计算`|当前服务器时间 - Timestamp|`，若超过允许范围（如5分钟），拒绝请求。
- **激活码校验**：查询数据库，确认激活码存在且状态为“未激活”，若已激活则返回错误。
- **防重放**：记录已使用的`(activation_code, nonce, timestamp)`组合，若重复则拒绝。
- **绑定**：将激活码状态更新为“已激活”，绑定`machine_code`，写入授权截止时间、功能位等。
- 构造响应明文：
  ```json
  {
    "result": "success",
    "authorized_until": "2026-12-31",
    "features": ["feature1", "feature2"],
    "nonce": Nonce_S   // 服务端生成的新随机数
  }
  ```
- 使用客户端公钥`PubC`（从请求中获得）对响应明文进行RSA-OAEP加密，得到密文`EncResp`。
- 通过HTTPS返回`EncResp`。

#### 3.3 客户端处理响应
- 使用客户端私钥`PriC`解密`EncResp`，得到响应明文。
- 校验`nonce`字段是否为服务端返回的新值，确保响应与本次请求对应。
- 若`result`为成功，将授权信息（截止时间、功能位）加密存储于本地（可使用机器码派生密钥加密），并记录绑定状态。

### 4. 后续启动验证
- 客户端每次启动时读取本地授权信息，若未过期则直接放行（离线可用）。
- 同时可选发起异步心跳请求（携带当前机器码和本地授权信息），服务端校验机器码是否匹配，若不一致则通知客户端重新激活。

### 5. 防破解增强
- 客户端代码使用虚拟化混淆，关键逻辑（如机器码生成、密钥存储、加解密）置于虚拟化保护中。
- 本地授权存储采用机器码派生密钥加密，防止授权文件跨机使用。
