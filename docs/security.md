# 安全模型

本篇说明 Sealium 的威胁假设、对应对策、最佳实践与**已知限制**，供你评估是否满足你的安全需求。

## 信任边界

```
[客户端机器]  ──不可信──►  [网络]  ──►  [激活服务]  ──►  [SQLite + 私钥]
  ▲ 内置公钥                              ▲ 持私钥
  └ 客户端代码可被逆向                     └ 信任根
```

- **客户端**：完全不可信——代码可被逆向、二进制可被篡改、运行环境可被调试。客户端只持有公钥。
- **网络**：不可信（中间人、被动抓包、主动篡改）。
- **服务端 + 私钥**：唯一信任根。私钥泄露 = 系统失守。

## 威胁与对策

| 威胁 | 对策 |
|---|---|
| **网络窃听激活码/机器码/授权** | 双层混合加密：RSA-4096 包 AES-256 密钥，AES-GCM 加密业务明文。整个负载端到端密文。 |
| **中间人篡改请求/响应** | AES-GCM 带认证 tag，篡改即解密失败；响应回显客户端 nonce，篡改/重放响应会被客户端拒绝。 |
| **伪造合法服务端响应** | 需服务端私钥才能解出请求里的 AES 密钥；客户端不持私钥，攻击者拿到公钥也无法解密他人请求或伪造可被客户端接受的响应。 |
| **重放激活请求** | 时间戳窗口（±300s，取自权威 API）+ `(code, nonce)` 去重。 |
| **客户端改本地时钟绕过过期** | 时间戳取自远程权威 API，非本地时钟。 |
| **枚举有效激活码** | 激活码 128 位随机，不可枚举；"不存在"与"已被占用"对外合并提示，关闭存在性枚举。 |
| **一码绑多机（并发抢绑）** | 绑定用单条 `WHERE status=UNUSED` 条件 UPDATE，并发仅一个命中（原子）。 |
| **激活码被复制到另一台机器** | 硬件指纹绑定；不同机器核心类不匹配 → 判异机拒绝。 |
| **spoof 工具篡改硬件序列号** | 多表面交叉验证（SMBIOS 固件表 + 磁盘 IOCTL + WMI），单层篡改即暴露；核心类占位符 → 分量缺失 → 核心门槛不过。 |
| **日志泄漏激活码/机器码** | 审计日志只记 SHA-256 短哈希（前 12 位）。 |
| **私钥文件被他人读取** | Linux 下落盘权限 `0600`；可加口令加密（`.env` `SEALIUM_SECURITY__PRIVATE_KEY_PASSPHRASE`，`SecretStr` 不回显明文）。Windows 下 `0600` 不生效（NTFS ACL 不同），私钥**必须**口令加密——但服务端默认部署在 Linux，故通常无需担心。 |
| **暴力刷激活接口** | 默认限流 60 req/60s/IP（`[rate_limit]`），超限 429。 |
| **接口结构泄露** | 生产（`[server] debug = false`）关闭 `/docs`/`/redoc`/`/openapi.json`。 |

## 最佳实践清单（生产部署）

> **默认部署方式：Linux + 反向代理（nginx 等）。** Sealium 服务端进程为明文 HTTP，设计为
> 置于反向代理 + TLS 之后，默认仅监听回环 `127.0.0.1`（同机反代转发）；反代跨机或容器时设
> `host = "0.0.0.0"` 并务必置于反代之后。服务端只在指纹比对（不采集硬件），故 Linux 部署完全
> 可用；仅客户端采集需 Windows。Windows 作服务端部署时 `0600` 权限语义不生效，私钥**必须**
> 口令加密（见 [服务端部署指南](server-guide.md)）。

- [ ] 私钥加口令加密落盘（`.env` `SEALIUM_SECURITY__PRIVATE_KEY_PASSPHRASE`）。
- [ ] 私钥文件权限 `0600`，所属用户最小权限。
- [ ] 服务置于反向代理后，启用 TLS + HSTS；`[server] host = "127.0.0.1"`。
- [ ] `[server] debug = false`。
- [ ] 限流开启并按实际调参（`[rate_limit]`）。
- [ ] 反代部署：把反代 IP 加入 `[server] trusted_proxies`（限流才按真实客户端 IP 分桶，HIGH-001），把对外域名加入 `[server] allowed_hosts`（启用 Host 头校验，LOW-006）。
- [ ] 部署前跑 `python -m sealium.server.config_cli check` 自检。
- [ ] 定期备份 SQLite（激活码是资产）。
- [ ] 多 worker 时注入共享防重放/限流后端（Redis）。
- [ ] 公钥随客户端分发，建立轮换机制。
- [ ] 客户端做反逆向/反调试/完整性校验（Sealium 不提供，属你的应用层）。

## 密码学参数

| 项 | 值 |
|---|---|
| RSA | 4096 位，OAEP(MGF1-SHA256)，e=65537 |
| AES | 256 位，GCM，12 字节 nonce，16 字节 tag |
| 激活码熵 | 128 位 |
| 临时 AES 密钥 | 每次请求新生成、用毕即弃 |
| 组件哈希 | SHA-256（每类硬件逐项，固定 pepper） |

均为当前业界推荐参数，无已知实际攻击。

## 已知限制（务必了解）

1. **客户端防破解不在 Sealium 范围**：Sealium 提供密码学绑定，但客户端 Python 代码本身可被
   逆向/篡改。攻击者可以：修改客户端跳过激活检查、patch `activate()` 直接返回 success、提取
   pepper 自行构造指纹。**对"高价值软件防盗版"，必须配合客户端加固**（打包成原生可执行、
   代码虚拟化/混淆、反调试、自身完整性校验）。Sealium 保证的是"激活码↔机器"的服务端绑定不可
   伪造，不是客户端不可破解。

2. **spoof_score 是软信号**：由客户端计算上报，恶意客户端可篡改为 0。硬防线是核心门槛（spoof
   常伴随核心占位符 → 分量缺失 → 门槛不过），但无法防御"精心伪造一份核心齐全的假指纹"——那
   需要服务端重采（纯在线架构做不到）。

3. **硬件采集仅 Windows**：客户端非 Windows 无法生成原生指纹。跨平台客户端需自定义 `machine_code_provider`。

4. **进程内有状态**：单实例下防重放/限流精确；多 worker 下为各进程独立（弱一致）。

5. **纯在线激活**：Sealium 不提供本地授权存储/离线校验。授权持久化与"何时再激活"由你的应用
   决定（见 [客户端集成 §7](client-guide.md#7-授权持久化你的应用负责)）。

6. **权威时间戳依赖第三方 API（隐私 / 可用性 / 完整性三项权衡）**：客户端时间戳取自第三方
   `https://aisenseapi.com/services/v1/timestamp`，非项目自有。集成前请评估以下三点：
   - **隐私**：每次 `activate()` 都会向该第三方发请求，向其泄漏「用户正在激活」这一行为以及
     客户端的公网 IP，可能被第三方用于用户画像 / 关联。
   - **可用性单点**：该 API 不可达时，所有客户端激活都会因 `ActivationError`（取时间戳失败）
     而中断。
   - **完整性**：请求虽走 TLS（`verify=True`），但未做证书 pinning、未禁重定向；若域名被 DNS
     劫持或 CA 被滥用，可返回伪造时间戳。注意时间戳**仅用于 ±300s 防重放窗口**，不参与授权
     截止判定（过期判断在服务端用服务端时钟），故伪造时间戳无法「延长授权」，最坏造成窗口内
     请求被拒（DoS）。

   若上述任一项不可接受，请注入你自己的 `timestamp_provider`（接口已开放，见
   [客户端集成 §6](client-guide.md#6-依赖注入测试定制)）：例如指向你自有的、多源冗余的时间服务，
   既消除向第三方的隐私泄漏与单点依赖，也便于你自行实施证书 pinning 或改用内网时间源。
7. **服务端默认部署在 Linux + 反向代理之后**：服务端进程为明文 HTTP，`0600` 文件权限在 Linux
   下生效、Windows 下不生效（NTFS ACL 不同）。默认部署在 Linux，故 `0600` 与"置于反代 + TLS 之后"
   均按 Linux 语义保证；若必须在 Windows 上跑服务端，私钥务必口令加密（见
   [服务端部署指南](server-guide.md)）。
