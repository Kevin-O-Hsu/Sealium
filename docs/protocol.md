# 加密与传输协议

Sealium 的激活通信采用**双层混合加密**：RSA 保护一次性 AES 密钥，AES-GCM 保护业务明文。
客户端只持有服务端**公钥**，因此即使客户端被完全逆向，攻击者也无法解密他人的激活请求
或伪造合法响应。

## 加密原语

| 用途 | 算法 | 参数 |
|---|---|---|
| 密钥包装 | RSA-OAEP | 4096 位，MGF1+SHA-256，公钥指数 65537 |
| 业务加密 | AES-GCM | 256 位密钥，12 字节 nonce，16 字节 tag |
| 激活码 | 随机 | 128 位（32 个十六进制字符） |
| nonce_C | 随机 | 16 字节（32 个十六进制字符） |

RSA-4096-OAEP 单次加密明文上限 = `key_bytes - 2*hash_len - 2` = `512 - 64 - 2` = **446 字节**，
足够装下 32 字节的 AES-256 密钥。

## 密钥分发

```
服务端持有：server_private.pem（私钥，绝不离开服务器）
客户端内置：server_public.pem（公钥，随客户端分发）
```

客户端**不需要、也不持有任何长期私钥**。响应用的是请求阶段客户端自己生成的临时 AES 密钥
（服务端从请求包里 RSA 解出来），用毕即弃。

## 请求数据包（客户端 → 服务端）

明文 JSON：
```json
{
  "activation_code": "32 位十六进制激活码",
  "machine_code": { "v": 1, "components": [...], "spoof": 0.0 },
  "timestamp": 1735689600,
  "nonce": "32 位十六进制客户端随机数"
}
```

> `machine_code` 是分量指纹对象（见 [硬件绑定原理](hardware-binding.md)）。

组装步骤：
1. 生成临时 AES-256 密钥 `K`（32 字节随机）与 AES nonce（12 字节随机）。
2. 用 AES-GCM(`K`) 加密请求明文 → `ciphertext` + `tag`。
3. 用服务端公钥 RSA-OAEP 加密 `K` → `encrypted_aes_key`（固定 512 字节）。
4. 拼接二进制包：

```
+-------------------+------------+---------------------+--------+
| encrypted_aes_key | aes_nonce  | ciphertext          | tag    |
| 512 字节          | 12 字节    | len(明文) 字节      | 16 字节|
+-------------------+------------+---------------------+--------+
```

通过 `POST /v1/activation` 发送，`Content-Type: application/octet-stream`。

> RSA 段长度按**实际加载私钥的位数**推导（`encryptor.key_size // 8`），而非硬编码 512——
> 兼容未来更换密钥长度。

## 响应数据包（服务端 → 客户端）

明文 JSON：
```json
{
  "result": "success",
  "authorized_until": "2026-12-31",
  "features": ["premium", "enterprise"],
  "nonce": "回显客户端的 nonce"
}
```

服务端用**步骤 2 中解出的同一个 `K`**、生成新的 AES nonce，AES-GCM 加密响应明文：

```
+------------+---------------------+--------+
| resp_nonce | resp_ciphertext     | tag    |
| 12 字节    | len(明文) 字节      | 16 字节|
+------------+---------------------+--------+
```

客户端用自己保留的 `K` 解密，并校验响应里的 `nonce` 与请求时发送的 `nonce_C` **严格相等**
——这同时保证：响应确实来自服务端（能用 `K` 解密即证明服务端持有私钥）、且未被重放/篡改。

## 防重放与时间戳

| 机制 | 说明 |
|---|---|
| **时间戳窗口** | `|服务端当前时间 - timestamp|` 超过 `TIME_STAMP_TOLERANCE_SECONDS`（默认 300 秒）即拒绝。客户端时间戳取自**权威远程 API**（`https://aisenseapi.com/services/v1/timestamp`），而非本地时钟，防客户端改系统时间绕过。 |
| **nonce 去重** | 服务端记录已使用的 `(activation_code, nonce)` 组合，重复提交直接拒绝。 |
| **响应 nonce 回显** | 见上，客户端校验回显 nonce。 |

防重放缓存为进程内、固定容量（`REPLAY_CACHE_SIZE`，默认 10000）LRU。多 worker 部署时各进程
独立——若需全局一致，注入共享后端。

## 错误响应也是加密的

除"解密前就失败"（包格式错误、RSA 段损坏 → 直接返回空体 400）外，所有业务错误（激活码无效、
已过期、时间戳超窗等）都**用同一 AES 密钥加密**后返回，错误码不会以明文泄漏：

```json
{ "result": "error", "error_msg": "激活码无效或已被使用", "nonce": null }
```

## 安全设计要点

- **不泄漏存在性**：激活码"不存在"与"已被他机占用"对外合并为同一条通用提示，关闭激活码存在性枚举攻击面。
- **日志脱敏**：服务端审计日志只记录激活码/机器码的**短哈希**（SHA-256 前 12 位），不记录原始值。
- **数据库权限**：SQLite 文件与私钥文件创建时权限收紧为 `0600`。
- **私钥可加口令**：`server_private.pem` 可用口令加密落盘，启动时通过 `SERVER_PRIVATE_KEY_PASSPHRASE` 解密。

## 传输层

应用层混合加密已让业务负载端到端保密。但**仍建议**在反向代理层终止 TLS（并启用 HSTS），
以隐藏时序、错误码、请求频次等元信息。Sealium 服务端进程默认是明文 HTTP，设计为置于
反向代理/防火墙之后，**不要裸暴露到公网**。
